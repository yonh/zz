# 交付审核清单

## 目的

在执行 `/review-delivery-completion` 时使用本清单，判断实现是否真正满足 active 任务要求，以及是否允许归档。

## 1. 需求完成情况
- [ ] 当前交付目标已真实达成
- [ ] active 文档中的每条成功标准都有证据
- [ ] 没有依赖 roadmap 中的能力来“伪装完成”当前目标

## 2. 代码现实与文档一致性
- [ ] `docs/active-work/<task>/` 仍与当前实现一致
- [ ] 文档中引用的文件、字段、配置、API、UI 元素都真实存在
- [ ] 配置示例与当前实现行为一致
- [ ] 当前实现没有依赖未记录的隐含假设

## 3. 行为与边界情况
- [ ] 主成功路径可用
- [ ] 失败路径已考虑
- [ ] 错误处理合理
- [ ] 目标范围之外的既有行为没有回归

## 4. 测试与验证
- [ ] 相关测试通过
- [ ] 必要时已完成手工验证
- [ ] 回归覆盖是明确的
- [ ] 已知限制被记录，而不是被隐藏

## 5. 范围纪律
- [ ] 当前实现没有悄悄扩展到 roadmap 范围
- [ ] 未完成的可选增强仍然被标记为 deferred
- [ ] 后续工作已按需拆到 `docs/roadmap/`

## 6. 归档结论
三选一：
- **Pass**：当前交付完成，可归档
- **Pass with Follow-up**：当前交付完成，但仍有后续项应留在 roadmap
- **Fail**：当前交付未完成，不能归档

## 7. 审核输出要求
一次完整的交付审核必须给出：
- 总体结论
- 阻塞问题（如有）
- 已记录的后续事项
- 是否允许执行 `/complete-and-archive-task`
