# /review-delivery-completion

目标：在实现完成后，用“独立审查者”视角判断当前任务是否真的完成，而不是只看代码是否已经写出来。

执行步骤：

1. 读取目标 active-work 目录：
   - `README.md`
   - `prd.md`
   - `tech-spec.md`
   - `implementation-plan.md`

2. 读取当前实现和测试证据：
   - 实际改动文件
   - 相关测试
   - 手工验证结果
   - 任何已知异常或未完成项

3. 用“需求完成审核”而不是“代码浏览”来审查：
   - 当前目标是否真正达成
   - 验收标准是否逐条满足
   - 是否存在未记录的行为变化
   - 是否存在异常路径未覆盖
   - 是否存在回归风险
   - 是否存在范围漂移（做多了或做偏了）

4. 必须特别检查：
   - 文档承诺的能力是否真的可用
   - 日志、错误处理、边界条件是否合理
   - 配置示例是否与当前实现一致
   - 当前实现是否偷偷依赖 roadmap 中尚未批准的能力

5. 输出审查结论时必须分级：
   - **Pass**：满足交付条件，可进入归档流程
   - **Pass with Follow-up**：主目标达成，但需要把剩余问题写入 roadmap
   - **Fail**：当前不满足交付条件，不能归档

6. 如果结论不是 Pass：
   - 明确阻塞项
   - 明确哪些是 bug
   - 明确哪些是文档失真
   - 明确哪些属于未来事项被错误混入当前实现

7. 最终输出：
   - 完成情况总评
   - 验收清单结果
   - 异常与风险列表
   - 是否允许执行 `/complete-and-archive-task`
